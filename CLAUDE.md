## Architecture — vgr-books is the Master

This project is the **authoritative source of truth** for all book production across the Publishing ecosystem. Everything flows toward it:

```
Source Projects          manuscripts/             vgr-books/
(editorial content)  →   (editor handoff)    →   registry.json  ← MASTER
artofgig/                .docx exchange           catalog
vgrtwitterbook/          transmittal              ISBNs
BreakingSmart/                                    stages
Ribbonfarm Archives/                              editions
Configurancy/                                         ↑
                                             Dashboard/ reads this
                                             (reporting view only)
```

**Roles:**
- **Source projects** — editorial authority. They build and maintain book content. They sync online editions here via `sync.py`. They update `registry.json` when book status changes.
- **`manuscripts/`** — editor handoff. .docx files live here when a book moves to beta. See `manuscripts/CLAUDE.md`.
- **`registry.json`** (this project) — production master. Lifecycle stages, editions, ISBNs, catalog display, transmittal data, ISBN block tracking. All agents working on any book should update this.
- **Dashboard** — reporting view. Reads `status.md` files for in-progress notes. Not authoritative.

**When agents from other projects need guidance**, point them here: `vgr-books/CLAUDE.md`.

## Local Development

```bash
# 1. Sync all books from source projects (first time, or after source changes)
python3 sync.py

# 2. Regenerate the catalog index (after editing registry.json)
python3 build_catalog.py

# 3. Start local server
bash serve.sh          # serves on http://localhost:8080
```

The site is a plain static site — no build step. Relative paths within each book's subdir work as-is.

## Canonical Source Authority

When a book has both an online version here and a print/ebook in Vellum development:

- **Before Vellum editing:** the source HTML/posts are authoritative; online site and docx are in sync.
- **Once Vellum editing begins:** the **edited .docx is the canonical source**. Any editorial changes flow from docx → online site, not the reverse.
- This applies immediately to **Mediocratopia**, which has a docx in `manuscripts/mediocratopia/` developing in parallel with its online site here.

Practical implication: when re-syncing a Ribbonfarm book after Vellum editing, regenerate the site from the exported docx content rather than from the raw posts.

## Registry

`registry.json` is the **single source of truth** for all book production tracking, edition management, and catalog display. It replaces the old `catalog_meta.json` + `status.md display` system.

**Never hand-edit `docs/index.html`** — it is generated by `build_catalog.py`.

### How it works

1. `registry.json` — hand-maintained (or agent-maintained). Contains every book with: identity fields, catalog display data (blurb, thumbnail, sort order), lifecycle stage, per-edition tracking (online/ebook/print), and transmittal form data.
2. A book card appears in the public catalog if any of its editions has `display: true`. Alpha-stage books should never have `display: true`.
3. `build_catalog.py` reads `registry.json`, filters to displayable books, resolves thumbnails, sorts by `sort_order`, and writes `docs/index.html`.

### Stage lifecycle

| Stage | Meaning | Catalog visible? |
|-------|---------|-----------------|
| `alpha` | Materials/idea in source project | No |
| `beta` | Working title assigned, transmittal submitted to editor | Only if `display: true` |
| `production` | Manuscript with editor, ISBNs assigned | Yes (if `display: true`) |
| `published` | Live, buy links active, Bowker updated | Yes (if `display: true`) |

### Edition tracks

- **online** — continuous deploy via `sync.py`. No ISBN. Separate from print/ebook pipeline.
- **ebook / print** — via Vellum → distributor. Requires ISBN. Governed by transmittal form.

### Catalog build command

```bash
python3 build_catalog.py          # regenerate docs/index.html
python3 build_catalog.py --dry-run  # preview output without writing
```

### Adding a book to the catalog

1. Add an entry to `registry.json` (copy an existing entry as template)
2. Set `editions.<type>.display: true` when the book reaches beta stage or beyond
3. Run `python3 build_catalog.py`

### Thumbnail convention by stage

| Stage | Thumbnail | Public catalog? |
|-------|-----------|----------------|
| `alpha` | `thumbnail: null` → `thumbnails/coming-soon.svg` (shared placeholder) | No |
| `beta` | `thumbnails/{id}-beta.jpg` — working mockup or early cover draft | Only if `display: true` |
| `production/published` (no slug) | `thumbnails/{id}.jpg` — final cover from source collateral | Yes |
| `production/published` (has slug) | Auto-resolved from `docs/{slug}/cover.jpg` etc. | Yes |

For source images, prefer purpose-built thumb variants from `collateral/` (e.g. `gpthumb.png`). Fall back to 75dpi or web-res variants. Never use 300dpi print-resolution files.

### Thumbnail resolution (in priority order)

1. `thumbnail` field in `registry.json` (if set and file exists under `docs/`)
2. `book_meta.json` in the source project folder — for books with no online slug. Image is copied to `docs/thumbnails/<id>.<ext>` on first catalog build. Run `python3 ../Dashboard/build.py` first to ensure `book_meta.json` files are current.
3. Convention files in `docs/<slug>/`: `cover.svg`, `cover.jpg`, `cover.png`, `thumbnail.jpg`, `thumbnail.png`
4. `docs/thumbnails/coming-soon.svg` (shared placeholder)

## URL Structure

| URL Pattern | Content |
|-------------|---------|
| `/` | Catalog index (generated) |
| `/{slug}/` | Book detail page (generated) |
| `/{slug}/read/` | Online reading experience (synced from source) |
| `/series/{series-slug}/` | Series home page (generated) |

Examples:
- `books.venkateshrao.com/artofgig-vol3/` — detail page for Art of Gig Vol 3
- `books.venkateshrao.com/artofgig-vol3/read/vol3_cover.html` — reading entry point
- `books.venkateshrao.com/series/art-of-gig/` — Art of Gig series page

Series slugs are auto-derived from the `series` field in `registry.json` (lowercased, spaces → hyphens).

## Catalog Design

- **Layout:** 5-column grid of compact cards (no blurb text on cards; cover + title + buy/read links only)
- **Featured book:** Random pick shown at top of page, selected by JS on page load. Blurb shown there.
- **Series handling:** No section headers on the catalog. Series books are grouped in order by `sort_order`. Individual book detail pages have a "More in this series" grid at the bottom.
- **Section ordering:** Controlled by `CATALOG_SECTIONS` list in `build_catalog.py` — a list of lists of book IDs. Reorder by editing that list.
- **Masthead:** Beige background, left-justified. Back link (← venkateshrao.com) above a dividing line, then "Books by Venkatesh Rao" h1.

## Adding Future Books

1. Add an entry to `registry.json` (copy an existing entry as template; set `stage`, `blurb`, `sort_order`)
2. Set `editions.<type>.display: true` when ready to show in catalog
3. Add the book's ID to `CATALOG_SECTIONS` in `build_catalog.py` in the desired position
4. If it has an online edition: add an entry to the `BOOKS` dict in `sync.py`, then run `python3 sync.py <slug>`
5. Run `python3 build_catalog.py` to regenerate all pages
6. If it has an online edition: create `deploy_to_vgr_books.sh` in the source project
7. Commit and push

> Do NOT edit `catalog_meta.json` — it is deprecated.
